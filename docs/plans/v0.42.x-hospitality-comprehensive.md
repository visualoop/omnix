## Guiding principles (from research)

1. **Menu items are NOT retail products.** They live only in the
   hospitality Orders workflow. A retail POS should never see a menu
   item as a sellable SKU. What DOES appear in retail POS is the
   raw inventory item (unga, sukuma leaves, cola bottle) OR the sale
   line the menu item produces AFTER cook.
2. **Every menu item has a Recipe.** Selling "Ugali Sukuma" deducts
   200g flour + 100g sukuma + 20g oil + 2 tomatoes + 0.5 onion. Recipes
   are a BOM (Bill of Materials); production.ts already implements FEFO
   deduction, we just wire it to menu items.
3. **Party size is required, named customer is optional.** Every table
   open must record cover count for reporting; only room-charging or
   loyalty flows need a named customer.
4. **Auto-fire to kitchen by default.** Modern POS (Toast, Square,
   Chowbus) fire items to KDS the moment they're added. Manual "Send
   to kitchen" is opt-in for slow-fine-dining review flow.
5. **Order status auto-derives from item status.** When every item is
   ready → order is ready → Order Board flips to READY column.
   When every item is served → order is served.
6. **Room service = one-click charge-to-folio.** Room-service order
   type + room number → auto-links to the room's open folio; on
   `served` the items post as folio charges.
7. **Room hierarchy**: Property → Floors → Rooms → Room Types.
   Housekeeping status is per-room. Reservations attach a guest to a
   room for a date range.

## Data model additions

### Migration 076 — recipes

```sql
CREATE TABLE IF NOT EXISTS recipes (
  id TEXT PRIMARY KEY,
  menu_item_id TEXT NOT NULL REFERENCES menu_items(id) ON DELETE CASCADE,
  yield_qty REAL NOT NULL DEFAULT 1,          -- one recipe → one plate by default
  notes TEXT,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE TABLE IF NOT EXISTS recipe_ingredients (
  id TEXT PRIMARY KEY,
  recipe_id TEXT NOT NULL REFERENCES recipes(id) ON DELETE CASCADE,
  product_id TEXT NOT NULL REFERENCES products(id),
  quantity REAL NOT NULL,
  unit TEXT NOT NULL DEFAULT 'g',             -- g / ml / pcs
  optional INTEGER NOT NULL DEFAULT 0,        -- e.g. lemon garnish
  sort_order INTEGER NOT NULL DEFAULT 0
);
CREATE UNIQUE INDEX IF NOT EXISTS idx_recipe_menu ON recipes(menu_item_id);
CREATE INDEX IF NOT EXISTS idx_recipe_ing_recipe ON recipe_ingredients(recipe_id);
```

### Migration 077 — floors (optional grouping)

```sql
CREATE TABLE IF NOT EXISTS floors (
  id TEXT PRIMARY KEY,
  branch_id TEXT,
  name TEXT NOT NULL,                          -- "Ground", "1st Floor"
  sort_order INTEGER NOT NULL DEFAULT 0,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
-- rooms.floor is already a free text column; add floor_id for structured pick
ALTER TABLE rooms ADD COLUMN floor_id TEXT REFERENCES floors(id);
```

- New "Recipe" tab on MenuItemDialog: pick ingredients from products
  with quantity + unit; running total costs computed live.
- When order item transitions to `served`: fetch its menu_item → recipe
  → for each ingredient, deduct FEFO from batches via
  production.ts::deductIngredients (rename existing production BOM logic

### v0.42.3 — Party size + room service

- `openOrder({ tableId, orderType, partySize })` requires partySize for
  dine_in + room_service.

  optional customer combobox.
- Room-service: room picker → auto-links to `guest_folios` row for
  that room's current booking. If no open folio → error toast.
- On order `served`, post its items to `folio_charges` for room-service.
- Runner view (in Hospitality hub) filters to
  `order_type = 'room_service' AND status = 'ready'` with room number

### v0.42.4 — KDS + Order Board content correctness

- KDS ticket shows: item name + photo (32px) + qty + modifiers + notes
  + allergen pills (rose when present) + elapsed time.

  new join to `menu_modifier_selections` (add if missing).
- Order Board shows: order number, table code (or Room 204 badge),
  guest name if present, party size, elapsed time. No item names on
  Order Board — that's for the customer-side, not detail.
- Customer name flows: on order open with named customer, denormalise
  into `hospitality_orders.customer_id`; queue query joins to
  `customers.name`.
- Real-time: bump refresh interval to 3s (currently 5s KDS, 2s queue);
  add a websocket-style store push if trivial to layer on top.

### v0.42.6 — Windows 11 snap layouts on custom titlebar

Two options; we pick the safer one:

**Option A (safe)**: use `tauri-plugin-snap-layout` community crate.
  - Cargo dep: `tauri-plugin-snap-layout = "0.3"`
  - Register the plugin in lib.rs
  - Attach `data-snap-layout` to our themed maximize button
  - Native Win32 overlay handles hover
  - Fallback on non-Win11 platforms (no-op)

**Option B (hand-roll)**: Write a small Rust plugin overriding
  `WM_NCHITTEST` to return `HTMAXBUTTON` when hovering our custom
  button. Needs `windows-sys` crate for message loop hooks. Higher
  maintenance cost.

**Decision**: try Option A first. If the crate is stale/broken, fall
back to Option B.
