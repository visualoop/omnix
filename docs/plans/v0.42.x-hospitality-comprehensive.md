## Guiding principles (from research)

1. **Menu items are NOT retail products.** They live only in the
   hospitality Orders workflow. A retail POS should never see a menu
   item as a sellable SKU. What DOES appear in retail POS is the
   raw inventory item (unga, sukuma leaves, cola bottle) OR the sale
   line the menu item produces AFTER cook.
2. **Every menu item has a Recipe.** Selling "Ugali Sukuma" deducts
   200g flour + 100g sukuma + 20g oil + 2 tomatoes + 0.5 onion. Recipes
   are a BOM (Bill of Materials); production.ts already implements FEFO
   deduction, we just wire it to menu items.
3. **Party size is required, named customer is optional.** Every table
   open must record cover count for reporting; only room-charging or
   loyalty flows need a named customer.
4. **Auto-fire to kitchen by default.** Modern POS (Toast, Square,
   Chowbus) fire items to KDS the moment they're added. Manual "Send
   to kitchen" is opt-in for slow-fine-dining review flow.
5. **Order status auto-derives from item status.** When every item is
   ready → order is ready → Order Board flips to READY column.
   When every item is served → order is served.
6. **Room service = one-click charge-to-folio.** Room-service order
   type + room number → auto-links to the room's open folio; on
   `served` the items post as folio charges.
7. **Room hierarchy**: Property → Floors → Rooms → Room Types.
   Housekeeping status is per-room. Reservations attach a guest to a
   room for a date range.

## Data model additions

### Migration 076 — recipes

```sql
CREATE TABLE IF NOT EXISTS recipes (
  id TEXT PRIMARY KEY,
  menu_item_id TEXT NOT NULL REFERENCES menu_items(id) ON DELETE CASCADE,
  yield_qty REAL NOT NULL DEFAULT 1,          -- one recipe → one plate by default
  notes TEXT,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE TABLE IF NOT EXISTS recipe_ingredients (
  id TEXT PRIMARY KEY,
  recipe_id TEXT NOT NULL REFERENCES recipes(id) ON DELETE CASCADE,
  product_id TEXT NOT NULL REFERENCES products(id),
  quantity REAL NOT NULL,
  unit TEXT NOT NULL DEFAULT 'g',             -- g / ml / pcs
  optional INTEGER NOT NULL DEFAULT 0,        -- e.g. lemon garnish
  sort_order INTEGER NOT NULL DEFAULT 0
);
CREATE UNIQUE INDEX IF NOT EXISTS idx_recipe_menu ON recipes(menu_item_id);
CREATE INDEX IF NOT EXISTS idx_recipe_ing_recipe ON recipe_ingredients(recipe_id);
```

  maintenance cost.

**Decision**: try Option A first. If the crate is stale/broken, fall
back to Option B.

# Critical clarifications (2026-07-03, from user)

## Hospitality is a different beast from Dawa / Retail / Hardware

The trade modules split into two mental models:

**"Shelf" modules — Dawa, Retail, Hardware**
- Sell **discrete SKUs** off a shelf.
- Inventory tracks quantity per SKU per branch.
- A sale line deducts 1 unit of that SKU from stock.
- Everything Omnix already does — POS grid, product detail, stock take, PO
  receive, batches, expiry — was built for this world.

**"Composition" module — Hospitality**
- Sells **menu items** that are recipes/compositions of ingredients.
- **The menu item itself has no stock**. Selling "Ugali Sukuma" doesn't
  deplete a "1 × Ugali Sukuma" — it deducts 200 g flour + 100 g sukuma
  + 20 g oil, etc., from raw-ingredient stock (which is normal inventory).
- Availability is derived, not stored: max servings = min over
  ingredients of `floor(stock / recipe_qty)` — the bottleneck.
- 86 (make unavailable) is a **soft, temporary** hide — the item still
  exists, its recipe still exists, its ingredients still have stock;
  the kitchen has decided not to sell it right now.
- Rooms are not stock either — they have status
  (available / occupied / dirty / maintenance) and folios (running tabs).
- Recipes bridge the two worlds: menu-item sale → recipe lookup →
  ingredient FEFO deduction. `planRecipeConsumption` (already wired) is
  the exact glue.

**Implications for v0.43.x tasks that were being over-generalised**:

1. **Menu-item detail page must NOT show "stock quantity"**. It shows:
   - Availability strip (max servings from bottleneck ingredient).
   - Recipe (the composition).
   - Sales trend (how many were ordered, not how many "in stock").
   - Cost history (recipe cost changes as ingredient prices change).
2. **"86" is UI-level unavailability**, distinct from `active=0`. Active
   means "retired forever, hide from menus". 86 means "we've run out
   tonight, back tomorrow". Separate columns, separate flows.
3. **Menu pagination + search** is over `menu_items` (not `products`).
   POS product-grid pagination already exists and correctly excludes
   `kind='menu_item'` from v0.42.1.
4. **The visual recipe builder is the primary inventory-adjacent flow
   for a menu item** — it's how the operator connects a menu item
   ("Ugali Sukuma") to its raw ingredients (unga, sukuma, oil, salt).
   Adding an ingredient in the builder is picking a product from
   `products` (`kind='physical'`), never another menu item.

## Release strategy — build everything, tag once

- No intermediate version bumps between v0.43.0 and v0.43.5.
- Build all of the v0.43.x scope in one push, verify once at the end,
  then bump to v0.43.0 and tag/push a single release.
- Only exception: if a change is a critical hotfix (e.g. tsc-blocking
  regression discovered mid-flight) it may bounce out as a v0.42.6+
  patch before rejoining the v0.43.x work.

# v0.50.x — Equipment layer for the Hardware & Equipment module

Turns the Hardware module into a lightweight equipment **Dealer Management
System (DMS)**: sell / warranty / service / rent serialized machines
(generators, excavators, mixers, plant, vehicles) alongside ordinary
hardware supplies — all on the existing Core (inventory, POS, customers,
credit, GL, eTIMS).

## Research basis (equipment DMS norms)
The unit record is the spine — **one serial = one unit** — and it follows
the machine through its whole life: purchase → sale → warranty → service →
rental → resale. Warranty start/end tie to the sale. Service orders read
the unit's history + warranty; parts come from the same inventory. Rental
reuses the unit. Everything posts to one GL (no reconciliation). Sources:
Suite Engine/RPM, Acumatica Equipment Mgmt, HBS, XAPT/NAXT.

## Architecture decision
**Extend Hardware + Core — no new module.** Serial tracking is a Core
inventory capability (a per-product flag + a `equipment_units` registry);
warranty + service + fleet UI live in the Hardware & Equipment hub. Same
customers, POS, credit, GL, eTIMS. The `products.kind` CHECK stays
`physical` (equipment IS physical stock); serial tracking is an overlay
flag, so no CHECK-constraint rebuild.

## Data model (migration 091)
- `products`: `+ tracked_by_serial INTEGER DEFAULT 0`, `+ warranty_months
  INTEGER`, `+ specs_json TEXT` (catalog-level specs: make, model,
  category, engine power kW/HP, displacement, fuel, operating weight,
  rating e.g. kVA/m³/kg).
- `equipment_units` — the registry: id, product_id, serial_number (unique),
  engine_number, chassis_number, year_of_manufacture, condition
  (new/used/refurbished), status (in_stock/reserved/sold/rented/in_service/
  written_off), acquisition_cost, acquired_at, branch_id, meter_value,
  meter_unit (hours/km), sale_id, customer_id, sold_at, warranty_months,
  warranty_start, warranty_expiry, specs_json (unit override), notes.
- (Phase 2) `service_jobs` + `service_job_parts` + `service_job_labour`.

## Phase 1 — Units + specs + warranty (v0.50.0) — THIS PLAN
Serialized units, spec fields, receive-by-serial, sell-a-specific-unit,
per-unit warranty auto-computed from the sale, warranty lookup by serial,
warranty-expiry alerts. Unblocks selling machines with full traceability.

Tasks: EQ-0 … EQ-9 (see task list).

## Phase 2 — Service & parts / workshop (v0.51.0) — roadmap
