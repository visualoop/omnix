# Omnix ‚Äî Salon / Spa Module (Sanaa) ‚Äî Comprehensive Plan

A new trade module (peer of Dawa / Hospitality / Hardware) for hair salons,
barbershops, nail bars, spas and beauty parlours. It is genuinely different
from retail: the unit of work is a **timed appointment with a named staff
member**, not a stock sale ‚Äî so it earns its own module rather than a Retail
toggle. It reuses Core throughout (customers = clients, POS, products, GL,
payments, eTIMS).

Working module name: **Sanaa** (Swahili "craft/art") ‚Äî matches the Dawa
naming energy. Internal id: `salon`.

## Industry coverage target (from Zenoti/Pabau/MioSalon/Vagaro/Booksy norms)
Booking + staff scheduling ¬∑ POS checkout ¬∑ client CRM + history ¬∑ staff
commissions + performance ¬∑ retail + back-bar inventory ¬∑ packages /
memberships ¬∑ reminders + no-show tracking ¬∑ reporting. (Client-facing
online booking is internet-dependent ‚Üí deferred; in-salon booking is core.)

## Architecture
- **Module:** new `salon` module + `sanaa` variant (follows the dawa/hardware
  pattern). Reuses Core: `customers` (clients), POS/`completeSale`, `products`
  (retail + back-bar), GL, payments, eTIMS.
- **Staff:** reuse `employees` where possible; a salon staff row adds
  services-they-perform (skills) + commission defaults + working hours.
- **Services are non-stock sale lines.** Key decision (see below).

## Data model
- `salon_services` ‚Äî id, name, category (hair/nails/massage/facial/barber‚Ä¶),
  duration_min, price, commission_pct, requires_room, color, active, notes.
- `salon_staff` ‚Äî id, employee_id/user_id, display_name, color, active,
  commission_default_pct.
- `salon_staff_services` ‚Äî staff_id √ó service_id (skills matrix).
- `salon_staff_hours` ‚Äî staff_id, weekday, start_time, end_time (+ day-off
  exceptions table `salon_staff_timeoff`).
- `salon_resources` ‚Äî id, name, type (room/chair/bed), active (optional in A).
- `salon_appointments` ‚Äî id, appt_number, client_id, staff_id, resource_id,
  starts_at, ends_at, status (booked/confirmed/checked_in/in_service/
  completed/no_show/cancelled), notes, sale_id, total, created_at.
- `salon_appointment_services` ‚Äî appt_id, service_id, staff_id, price,
  duration_min, commission_amount.
- `salon_client_profiles` ‚Äî client_id, preferences, allergies, formulas
  (e.g. hair-colour recipes), notes.
- `salon_packages` / `client_packages` ‚Äî prepaid session bundles +
  memberships (sessions_remaining, expires_at, purchase_sale_id).
- `salon_commissions` ‚Äî accrual ledger (staff_id, appt_id/sale_id, kind
  service|retail, base, pct, amount, period) ‚Üí feeds payroll.
- Back-bar consumption reuses `stock_movements` (type adjustment, ref
  `salon_service`).

## Key technical decisions
1. **Service checkout line.** `completeSale` today needs a `product_id` and
   runs a stock preflight on physical lines (hospitality menu lines already
   skip it via `menu_item_id`). Decision: **generalize the non-stock line
   path** ‚Äî add a `service_id` line kind that skips batch stock, so salon
   services check out through the normal POS ‚Üí GL ‚Üí eTIMS ‚Üí commission flow.
2. **Commission engine.** Accrue on checkout: service commission (per
   service pct or staff default) + retail commission; write to
   `salon_commissions`; surface in a staff-performance report and feed
   payroll. Reuse the hardware commission patterns where sensible.
3. **Calendar is the centrepiece.** Day view = time-rows √ó staff-columns
   grid; booking dialog with **slot conflict detection** (staff + resource
   busy check) and service-duration auto-fill of the end time. Week view in
   Phase B.
4. **Offline-first.** All booking/checkout is local. SMS reminders +
   client-facing online booking are internet features ‚Üí Phase B / optional,
   behind connectivity checks.

## Phases

### Salon Phase A ‚Äî Booking & checkout MVP (v0.57.0)  [BUILD FIRST]
Services catalog ¬∑ staff + skills + working hours ¬∑ day-view calendar +
booking with conflict detection + status flow (check-in ‚Üí in-service ‚Üí
complete) ¬∑ checkout to POS (services + retail + tips) ¬∑ staff commission
accrual ¬∑ client salon profile + visit history ¬∑ Salon hub + `sanaa` variant.

### Salon Phase B ‚Äî Growth (v0.58.0)
Packages / memberships (prepaid sessions) ¬∑ back-bar consumption ¬∑ week +
resource calendar ¬∑ no-show tracking + rebooking prompts ¬∑ SMS reminders
(connectivity-gated) ¬∑ staff-performance + rebooking-rate + retail-attach
reports ¬∑ client formulas/photos.

## Ship rhythm
Phase A is itself large; it will ship as one or two atomic tags
(A1: data + services/staff/skills + commissions engine; A2: calendar +
booking + checkout). Each through tsc + vitest + audit + website-build gate.
