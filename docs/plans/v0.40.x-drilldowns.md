# v0.40.x — Drill-down detail pages

## Problem

User creates a "Main Hall" area in Hospitality → nothing to click, nothing
to open, no way to see the tables in that area or manage the area itself.
Same story across the app: brands, categories, room types, floors, doctors,
customers, patients — most entity list rows are terminal, not navigable.

The `audit-drilldowns.mjs` script (v0.40.0-plan) finds 74 non-clickable
list rows. Filtering out legitimately-non-navigable sub-items (sale items,
invoice items, folio charges — data rows, not entities), the real gap is
about 25 rows across ~15 entity types.

## Approach

For every navigable entity, add:
1. **Detail route** — `/entity-plural/:id`
2. **Detail page** — masthead (name, meta), tabs (Overview / related entities / activity), actions
3. **Row click handler** in the list page — `onClick={() => navigate(`/entity-plural/${row.id}`)}`
4. **Back button** on the detail page pointing to the list

Keep the pattern uniform. Reuse existing detail-page patterns from
`customer-detail.tsx`, `product-detail.tsx`, `supplier-detail.tsx`.

## Scope — Hospitality first

The user's example was Hospitality, so start there:

| Entity | Route | Detail shows |
|---|---|---|
| Dining area | `/hospitality/areas/:id` | Tables in this area, active orders, staff assigned |
| Dining table | `/hospitality/tables/:id` | Current status, active order, upcoming reservations, transaction history |
| Kitchen station | `/hospitality/stations/:id` | Menu items routed here, staff on shift, today's ticket load |
| Room type | `/hospitality/room-types/:id` | Rooms of this type, occupancy rate, rate-plan history |
| Room | `/hospitality/rooms/:id` | Current occupant + booking, housekeeping status, past bookings, revenue |
| Booking | `/hospitality/bookings/:id` | Guest, dates, folio charges, payment status |
| Guest | `/hospitality/guests/:id` | Bookings history, folio balances, notes |
| Menu item | `/hospitality/menu-items/:id` | Recipe, cost, sales history, availability toggle |

## Non-scope

- Sale items / invoice items / folio charges → data rows, not entities.
  A user reading an invoice doesn't need to "drill into" line 3 —
  the line IS the drill-down.
- Audit log entries → each entry IS the detail, no deeper view.
- Notification centre entries → each entry is a link to somewhere else,
  not its own detail.
- Held sales, reservations pending → open in a dialog, not a route.
