## Critical bugs (data / crash / compliance)

### C1. Hardware module crashes: "no such column: quote_number"

**Root cause (verified):** migrations `018_invoicing.sql` and `031_hardware.sql`
both do `CREATE TABLE IF NOT EXISTS quotations` and `quotation_items` with
**different columns**. 018 runs first (numerically earlier) → its schema wins.
The `IF NOT EXISTS` clause on 031 silently skips because a table with that
name already exists. Result:
- `quotations` gets columns from 018 (`quotation_number`, `customer_name`,
  `issue_date`, `discount_amount`, `subtotal`, `total`, `user_id`)
- Missing (that 031 wanted): `quote_number`, `salesperson_id`,
  `converted_sale_id`, `discount`, `branch_id` — no wait 018 has branch_id.
- `quotation_items` gets columns from 018 (`description`, `unit`,
  `discount_amount`, `sort_order`)
- Missing (that 031 wanted): `name`, `uom`, `discount`

**Every hardware query that fails now:**
- `INSERT INTO quotations (id, quote_number, branch_id, customer_id, status, valid_until, subtotal, discount, tax_amount, total, salesperson_id, notes, created_by)`
- `SELECT id, quote_number, customer_id, status, total, valid_until, converted_sale_id, created_at FROM quotations`
- `INSERT INTO quotation_items (id, quotation_id, product_id, name, uom, quantity, unit_price, discount, tax_rate, line_total)`
- `SELECT customer_id, status, discount FROM quotations`
- `SELECT product_id, name, quantity, unit_price, discount, tax_rate, line_total FROM quotation_items`

**Fix approach:**
1. **New migration 073_hardware_quotation_columns.sql** — ALTER TABLE quotations
   ADD COLUMN for the hardware-only fields that don't collide:
   `salesperson_id TEXT`, `converted_sale_id TEXT`. Same for quotation_items:
   ADD COLUMN `uom TEXT`.
2. **Rewrite `src/services/hardware.ts`** SQL to use 018 column names:
   `quote_number` → `quotation_number`, `discount` → `discount_amount`,
   `name` → `description`. Provide the NOT-NULL fields 018 requires
   (`customer_name`, `valid_until`, `user_id`, `issue_date`).
3. **New test `tests/hardware.spec.ts`** — seed product + customer + user,
   `createQuote()` then `getQuote()`, verify the round-trip.
4. **Delete or comment 031's CREATE TABLE stanzas** with a note explaining
   why (leave delivery_notes / commissions / contractor tables intact).

**Impact:** Fixes the crash the user reported. Contained to one service file
+ one migration + one test.

### C2. POS silently dispenses expired batches

**Root cause (verified):** every stock-decrement site uses either:
- `ORDER BY received_at ASC` (sales.ts:229 — FIFO by receive date)
- No `WHERE` clause on expiry

Files with the same defect:
- `src/services/sales.ts:229` — the till itself
- `src/services/operations.ts:189` — deliveries
- `src/services/production.ts:287` — recipe ingredient consumption
- `src/services/stock-transfers.ts:147` — branch transfers

For a pharmacy this is a compliance and patient-safety issue. Product-detail
already flags expired batches in red, but nothing stops them being dispensed.

**Fix approach:**
1. Change all four sites' FIFO SQL to FEFO:
   ```sql
   WHERE product_id = ?1
     AND quantity > 0
     AND (expiry_date IS NULL OR expiry_date > date('now'))
   ORDER BY expiry_date ASC NULLS LAST, received_at ASC
   ```
2. Update the shortage calculation in sales.ts:114 to exclude expired
   batches from `available` (`WHERE ... AND (expiry_date IS NULL OR
   expiry_date > date('now'))`).
3. Add `tests/fefo.spec.ts` — seed a product with 3 batches (one expired,
   one expiring in 7 days, one expiring in 6 months). Verify: shortage
   check excludes the expired batch; sale picks the 7-day batch first
   (FEFO), not the expired one.

**Impact:** Real safety fix for a Kenyan pharmacy. Small surface change,
mid-risk of regressions — hence the dedicated test.

## High-severity UX gaps

### H1. Hardware landing page is a dead end (7 orphan routes)

The user just navigates to `/hardware` and sees no way to reach:
- `/hardware/dashboard`
- `/hardware/quotations`
- `/hardware/delivery-notes`
- `/hardware/accounts` (contractor accounts + credit / aging)
- `/hardware/commissions`

Only the Cmd+K palette knows they exist.

**Fix approach:** Add a "Hardware operations" grid on `/hardware` with 6 stat
cards each linking to a sub-route. Same visual language as
`/retail-dashboard`'s existing card grid so the module stays consistent.

## Medium-severity UX

### M1. Purchase-orders supplier picker — no way to create supplier inline

**Current state:** `/purchase-orders/new` uses a bare `<Select>` bound to the
suppliers list. If the supplier isn't there, user leaves, creates,
comes back, and starts the PO over. Common friction.

**Fix approach:**
- Swap the `<Select>` for a Combobox (`cmdk` — already a dep, used in the
  command palette).
- Typeahead against `listSuppliers()`. Bottom row of the popover is
  "+ Create supplier '<typed text>'".
- Click → navigate to `/suppliers/new?prefill=<name>&returnTo=/purchase-orders/new`.
- `suppliers-new` reads `?prefill=` and populates the name field.
- After save, navigate to `returnTo` with `?supplier=<newId>` so PO form
  auto-selects.
- Apply the same pattern to other supplier pickers: goods-receipt-new,
  debit-notes, PO detail edit.

### M2. POS low-stock badge — verify + polish

User: "thats when a product is low quantity from the pos it shows". Need to:
1. Read `src/pages/pos-sale.tsx` + the product-tile component.
2. Confirm a badge appears when `qty ≤ reorder_level`.
3. Confirm a distinct red badge when `qty = 0` (already blocked by shortage
   check, but visible earlier is better).
4. Same badge on the search-results row.

Non-blocking; sale will still error out on shortage.

## Polish / theming (v0.38.0)

### P1. Light mode too white, dark mode too black

User's complaint is legitimate — pure `#FFF` and pure `#000` are unforgiving
on the eyes. AGENTS.md rule says "Linear/Notion-grade desktop app", and
neither Linear nor Notion uses pure white/black.

**Approach:**
1. Read `src/index.css` / `tailwind.config.js` — find the CSS variables
   driving `--background`, `--foreground`, `--card`, `--muted`, `--border`.
2. Light mode: `--background` from `hsl(0 0% 100%)` (pure white) to
   `hsl(48 20% 98%)` (warm off-white, matches Notion cream).
3. Dark mode: `--background` from `hsl(0 0% 4%)` (near-black) to
   `hsl(220 15% 8%)` (slate-tinted dark, matches Linear).
4. Adjust `--card` / `--muted` / `--border` in the same hue family so
   surfaces read as one system.
5. Manually spot-check high-density surfaces (POS tile grid, KDS, tables)
   for contrast regressions.

**Ship as its own release** so the theming diff is reviewable in isolation.
