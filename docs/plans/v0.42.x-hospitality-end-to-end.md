# v0.42.x — Hospitality end-to-end: KDS / Order Board / room service / menu images

## Diagnosis (from code inspection)

### Order flow today

```
[Orders tab] newOrder() ──► hospitality_orders (status=open)
                              │
[Orders tab] addItem() ──────┴──► hospitality_order_items (status=new)
                                    │
                                    │  ← manual click required
                                    ▼
[Orders tab] send() ─── UPDATE items SET status='sent', sent_at=now()
                        UPDATE order   SET status='sent'
                                    │
                                    ▼
                        KDS query: items WHERE status IN ('sent','preparing')
                                    │
[KDS] bumpTicket() ─── UPDATE items SET status='ready', ready_at=now()
                                    │
                                    │  ← MISSING: order status stays 'sent'
                                    ▼
                        Order Board splits by ORDER status:
                          PREPARING = open / sent / preparing
                          READY     = ready
```

### Two real bugs

**Bug A — order status never advances to `ready`.** Even when every item is
`ready`, the parent `hospitality_orders.status` stays `sent`. The Order
Board relies on the ORDER status to move to READY column, so orders
never appear on the customer-facing READY side. Guests never see their
order is ready.

**Bug B — "Send to kitchen" is manual + easy to miss.** Staff take orders,
add items, and forget to hit Send. Items stay `new`, KDS shows nothing.
Modern KDS systems auto-fire on order confirmation.

### Missing features

- **Menu-item photos** — schema has no image column; UI has no upload.
  Kitchen staff and customers benefit from photos (ordering confidence
  + kitchen accuracy).
- **Room-service order routing** — Orders tab has a `room_service` order
  type but the flow doesn't attach a guest folio. Room-service items
  need to charge back to the room folio automatically.
- **Real-time refresh** — KDS + Order Board poll every 2-5s. On busy
  services this is fine, but tickets can lag by up to a poll interval.
- **Modifiers + allergies** — schema has `menu_modifiers` but the UI
  doesn't surface them; the KDS never shows allergen flags.

## Fixes to ship

### v0.42.0 — Order status auto-advance
- When every non-voided item on an order is `ready`, mark the order
  `ready` too (via a `refreshOrderStatus` helper called after each
  bump).
- When every item is `served`, mark the order `served`.
- Backfill: existing orders don't get retro-treatment — just applies
  going forward.

### v0.42.1 — Auto-fire on order confirm
- Split the "Send to kitchen" button into two modes:
  1. **Auto-fire** (default) — items go straight to `sent` on add.
     Order status flips to `sent` on first item. This is the industry
     default; every mainstream POS/KDS works this way.
  2. **Manual** (opt-in via settings) — the current behaviour, for
     businesses that want a "review before send" workflow (e.g. a
     server checks the order with the guest before firing).
- Setting: `hospitality.auto_fire = 'on' | 'off'`. Default `on`.

### v0.42.2 — Menu-item photos
- Migration adds `menu_items.image_path TEXT` (nullable).
- MenuItemDialog gets an image upload input. Photos stored as data
  URLs (same pattern as business logo). Max 1MB, auto-shrink to 400px.
- Photos render on:
  - Order-taking tile (Orders tab menu grid) — small square left of name
  - KDS ticket — small 32px square next to item name so the chef
    spots the item quickly
  - Order Board — no photos (keeps the wall screen readable at
    distance)

### v0.42.3 — Room-service ordering closes the loop
- When `openOrder({ orderType: 'room_service' })`, require a `room_id`
  and auto-link the order to the currently-open folio for that room.
- On send-to-kitchen, the KDS shows a Room badge (e.g. "Room 204")
  instead of Table code so the runner knows where to deliver.
- On `served`, the order's line items post to the guest folio as
  charges (existing `postOrderToRoomFolio` service, wire it into the
  served-transition).

### v0.42.4 — Allergen + modifier surface
- Fetch modifiers on order items into the KDS query (already stored
  in `hospitality_order_items.modifier_total` but no modifier detail
  is fetched).
- Add an allergen column to `menu_items` (`allergens TEXT` — comma
  list). Render red pill on KDS ticket when the item has allergens
  matching the guest's profile if a customer is attached.

## Non-scope

- QR-code table ordering (customer scans, orders from phone) — big
  product surface; separate release
- Third-party delivery integration (Doordash / Uber Eats) — not
  applicable in Kenya market yet
- Video prep instructions on KDS — nice but low-ROI
- AI-suggested prep-time overrides — v0.43+
