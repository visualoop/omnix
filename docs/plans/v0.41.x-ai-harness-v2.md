# v0.41.x — AI Harness v2 (agentic + tool-calling)

## Current state (v0.40.5)

Our AI layer under `src/services/ai/`:
- **Config + auth** — provider list, feature toggles, API keys stored in
  the settings table. Multi-provider (OpenAI, Anthropic, Groq, OpenRouter).
- **Tasks** — one-shot AI functions in `services/ai/tasks/*.ts`:
  invoice OCR, natural-language search, forecasting, drug enrichment,
  docs Q&A, import normalisation, etimseExplanation. Each function
  assembles a prompt, calls the LLM, parses the response.
- **Tools** — `services/ai/tools.ts` defines `buildAssistantTools(ctx)`
  returning ~10 tools (get_low_stock, get_expiring, etc.) as OpenAI-style
  function schemas.
- **Streaming** — `services/ai/stream.ts` handles SSE + tool-call parsing.
- **Retry** — `services/ai/retry.ts` handles token-idle + retry policy.
- **Compression** — `services/ai/compression.ts` shrinks conversation
  when it approaches the model's context window.
- **Cache** — `services/ai/cache.ts` deduplicates identical requests.
- **Redact** — `services/ai/redact.ts` strips PII before hitting a
  cloud model (privacy tier).
- **Audit** — every AI call logged to `ai_calls` with tokens, cost,
  latency, privacy tier, source page.

## What's missing to make it truly agentic

Compared to opencode / Claude Code / cursor:

1. **Multi-turn tool loop** — right now a task calls the LLM once and
   parses the result. Real agentic behaviour is: LLM emits tool call →
   we execute → feed result back → LLM decides next step → repeat until
   the model produces a final text answer or we hit a max-iteration
   guard. Our streaming layer has the primitive but no orchestrator.

2. **Agent roles / modes** — opencode has `build`, `plan`, `general`
   agents. Each sees a different tool subset. We need at least two
   for Omnix:
   - `read` — safe browsing (queries + reports, no writes)
   - `write` — full access (can post journal entries, create products,
     void sales, etc.) — needs approval gate for destructive ops
   - Optional `plan` — dry-run mode that describes what it would do
     without touching data

3. **Tool registry with permission ruleset** — currently tools are
   built ad-hoc per call site. A registry with typed inputs, outputs,
   permission tags (`read` / `write` / `destructive`), and a rule
   engine ("write-tier tools require operator approval") would give us
   real safety.

4. **Sub-agents** — a planner agent that spawns a specialist
   ("look up expiring items and draft a discount promotion")
   composed of two agent turns instead of one bloated prompt.

5. **Approval prompts** — when a write-tier tool is about to run,
   surface a card in the UI showing what will happen. User approves
   or rejects. Rejection feeds "denied" back to the model so it can
   try a different approach.

6. **Streaming tool-run cards** — while the LLM is running tools,
   show each tool invocation as a card in the assistant reply:
   input, running, result (or error). Users see what the agent is
   doing, not a black box.

7. **Error recovery** — when a tool fails, feed the error back to
   the model so it can self-correct (up to N attempts) before
   giving up. Currently a tool error bubbles up as an assistant error.

8. **Session state** — opencode has per-session tool state
   (last query, previous cart, etc.). We should have this via a
   session-scoped context object passed to every tool invocation.

9. **Telemetry hook per tool** — every tool run should emit a
   telemetry event with duration, success/fail, args, result size.
   Feeds cost + quality dashboards.

## Design

### New modules

```
src/services/ai/
  agents/
    index.ts           — agent registry
    read-agent.ts      — read-only agent
    write-agent.ts     — full-access agent
    plan-agent.ts      — dry-run agent
  runtime/
    agent-loop.ts      — the multi-turn tool loop
    tool-runtime.ts    — permission-checked tool executor
    approval.ts        — approval-prompt orchestration
    session-context.ts — per-session state passed to tools
  tools/
    registry.ts        — typed ToolDefinition + ToolRegistry
    read/*.ts          — read-tier tools (queries, reports)
    write/*.ts         — write-tier tools (create product, void sale)
    system/*.ts        — meta tools (search, plan, todo)
```

### `ToolDefinition` shape

```ts
interface ToolDefinition<Input, Output> {
  name: string;
  description: string;
  input: ZodSchema<Input>;
  output: ZodSchema<Output>;
  tier: "read" | "write" | "destructive";
  /** UI label + icon for the tool-run card */
  ui: { label: string; icon?: string };
  /** Actual work. Receives session ctx + parsed input. */
  execute: (ctx: SessionContext, input: Input) => Promise<Output>;
}
```

### Agent loop pseudocode

```ts
async function runAgent(agent, initialPrompt) {
  const messages = [systemPrompt(agent), userMessage(initialPrompt)];
  const tools = registry.forAgent(agent);
  for (let iteration = 0; iteration < MAX_ITERATIONS; iteration++) {
    const response = await llm.streamChat({ model, messages, tools });
    if (response.type === "text") {
      messages.push(response);
      break;    // final answer
    }
    if (response.type === "tool_calls") {
      messages.push(response);
      for (const call of response.calls) {
        const tool = registry.get(call.name);
        if (tool.tier === "write" && !await approve(tool, call.args)) {
          messages.push(toolResult(call.id, { error: "user_denied" }));
          continue;
        }
        try {
          const result = await tool.execute(ctx, call.args);
          messages.push(toolResult(call.id, result));
        } catch (e) {
          messages.push(toolResult(call.id, { error: String(e) }));
        }
      }
      continue;   // let model decide next turn
    }
  }
}
```

## Ship order

Incremental so each step is testable and reversible:

| Version | Contents |
|---|---|
| **v0.41.0** | New tool registry + typed ToolDefinition. Move existing tools into `tools/read/*.ts` behind the registry. No behaviour change. |
| **v0.41.1** | Agent-loop runtime — multi-turn tool calling with max-iteration guard. `chat` endpoint switches to loop by default. |
| **v0.41.2** | Approval prompts for `write` and `destructive` tools. UI card in the AI panel; approval feeds back into the loop. |
| **v0.41.3** | Tool-run cards in the chat UI — show each tool invocation with input + result + duration. |
| **v0.41.4** | Session context passed to every tool (branch, user, active cart, etc.). Enables tools like `add_to_cart` that need current state. |
| **v0.41.5** | Sub-agent primitive — planner agent can delegate to a specialist. Introduces `plan-agent`. |
| **v0.41.6** | Error recovery — tool failures feed back to model for self-correction up to N times. |
| **v0.41.7** | Telemetry per tool run — hook into existing `ai_calls` audit but at tool granularity. |
| **v0.41.8** | Sample write-tier tools: `create_product`, `record_expense`, `void_sale`. Each requires approval. Real agentic value unlocks here. |

## Non-scope

- Multi-model routing beyond current provider set — we already have
  OpenAI/Anthropic/Groq/OpenRouter; adding more is separate work
- TUI / desktop / web split like opencode — Omnix already has its own
  window layer, we don't need to rebuild
- Effect-TS migration — opencode uses Effect but we don't need to
  refactor our stack for it; concepts are portable
- MCP integration — Model Context Protocol support is a valid v0.42+
  goal but out of scope here
