# Omnix — 99% Industry Coverage Program (v0.53+)

Senior-engineer program to push every module toward ~99% coverage of its
industry, and to turn cross-cutting features (serial tracking, warranty,
returns/RMA) into **Core capabilities available everywhere** — product
creation, product detail, POS — rather than being walled inside one module.

## Guiding architecture principle
**Capabilities, not a module per shop type.** A module exists only when the
*data model + workflow* are genuinely different (Pharmacy, Hospitality,
Salon). Everything else (minimart, electronics, boutique, auto-parts) is
Retail/Core + capability toggles.

## Key finding to fix first — two parallel serial systems
- `product_serials` (migration 065, `product-serials.ts`): generic
  electronics serials — **built but wired into NOTHING** (orphaned).
- `equipment_units` (migration 091, `equipment.ts`): richer (warranty
  dates, meter, service + rental linkage), **wired but hardware-gated**.

**Decision:** standardize on `equipment_units` as the single serialized-unit
registry; generalize its UI out of the `activeModule === "hardware"` gate so
retail/electronics/pro can use it; treat `product_serials` as legacy (kept
only for its existing tests). One serial system, everywhere.

## Per-module gap analysis → target 99%

### Core (inventory / POS / purchasing / accounting)
Built: batches, FEFO, stock takes, transfers, PO/GRN, GL, eTIMS, reports.
Gaps: **serial/unit visibility on the product-detail page**, **returns/RMA
as a first-class flow**, barcode-label printing per unit, bin/location.

### Retail (Soko)
Built: variants, price lists, loyalty, promotions, layby, special orders,
shrinkage, shelf labels.
Gaps: **serialized goods + warranty (currently hardware-gated)**, **RMA /
returns to supplier**, **gift cards / store credit**, **bundles/kits wired
to POS** (a `bundles.ts` exists — verify it's surfaced), hire-purchase.

### Electronics (new — target install next month)
= Retail + serialized units + warranty + repairs (service jobs) − rental.
Gaps: **electronics variant**, **variant-level serials/IMEI**, trade-in,
repairs relabelled from "service".

### Dawa (Pharmacy)
Built: prescriptions, interactions, controlled substances, cold chain, SHA,
licences, e-prescriptions, PPB. Near-complete.
Gaps: minor — insurance pre-auth batching, expiry/return-to-supplier.

### Hardware & Equipment
Built: quotes, contractor credit, delivery notes, serialized units, warranty,
service workshop, rental fleet. Near-complete.
Gaps: preventive-maintenance schedules (by meter/date), OEM warranty claims.

### Hospitality
Built: tables, KOT/kitchen, recipes, rooms, bookings, folios.
Gaps: split bills polish, channel-manager depth, table reservations UX.

### Salon / Spa (new — later)
Genuinely different → **its own module**: appointment calendar, service
catalog (time-based, not stock), staff rosters + commissions, client history.

## Phased roadmap (each phase = one atomic, verified, tagged release)

- **Phase 1 (v0.53.0) — Serial everywhere [THIS PHASE].** Un-gate the
  serialized-unit + warranty capability from hardware. `moduleTracksSerials()`
  helper drives the product-creation "Serial & Warranty" tab + POS
  unit-picker for retail/electronics/pro/hardware. Add a **Units tab to the
  product-detail page**. Module-aware labels.
- **Phase 2 (v0.54.0) — Electronics enablement [capability, not a branded variant].**
  Surfaced Units + Repairs tabs in the Retail hub and relaxed the equipment/
  service mutations from a hardware *entitlement* to permission-only, so a
  retail (or pro) install works end-to-end as an electronics shop: flag a
  product serial-tracked → receive units → sell with warranty → repair. A
  dedicated branded "Omnix Electronics" build (license prefix, masthead) is
  deferred as optional polish — functionally retail + capability = electronics.
  Follow-ups: variant-level IMEI (`equipment_units.variant_id`) + RMA statuses
  land with Phase 3.
- **Phase 3 (v0.55.0) — Returns / RMA + store credit.** First-class customer
  return → restock or quarantine, store-credit/gift-card ledger, supplier RMA.
- **Phase 4 (v0.56.0) — Retail depth.** Bundles/kits at POS, hire-purchase,
  gift cards at POS.
- **Phase 5 (v0.57.0) — Salon / Spa module.** Appointments, service catalog,
  staff commissions, client history.
- **Phase 6 (v0.58.0) — Preventive maintenance + module polish.** PM
  schedules (equipment), pharmacy return-to-supplier, hospitality split-bill
  polish.

Phases execute in order; each is planned into its own task list, built,
verified (tsc + vitest + audit + website build), and tagged.
